Csaba Hoch — <hh/214> <↑hh/213> (2009-05-11) 
No, I don't want "todo" to be a flag either. It _will be_ a meta
attribute; why should it be something else, too? Why isn't it enough,
if it is _only_ meta attribute? Heapkeeper is (or will be with the
"custom post summary feature") flexible enough so that we don't need
to convert a meta attribute into anything else in order to display it
in the required way.

Every information that belongs to a post should be described only in
one place in a post file. We may generate any number of lazy data
attributes for the Post from them, but the core information should be
as redundancy-free, as possible. If we had the "todo" tag both as a
meta attribute (with the attached meta text) and a flag or tag, the
post would be unnecessarily redundant.
Csaba Hoch — <hh/215> <↑hh/214> (2009-05-12) 
I have been thinking about this, and meta attribute is as good as a
flag, even if we don't have the "custom post summary" feature! If we
want "todo" to be displayed specially in the post summary, we have to
hard-code displaying posts with "todo" differently in either case --
it is just that if we have a flag, we have to hard-code "if the post
has a 'todo' flag, do X", while if it is only a meta attribute, we
have to hard-code "if the post has meta attribute 'todo', do X".

I also have been thinking about why do you insist on having a flag for
'todo'. If it is sheer misunderstanding, I hope this thread helps. But
I realized you may have another concern: performance. You don't want
to parse all posts all the time when only generating an index. I have
to assure you, though, that having a "todo" flag is not the right
solution to this performance issue: there are much better and much more
generic solutions. (I'll talk about one if you are curious.)
