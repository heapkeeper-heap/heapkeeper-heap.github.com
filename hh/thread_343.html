Previously I argued that all todos should be in our todo file.
I changed my mind: I think we need to document each function
separately and locally whether they have tests and documentation. This
would be especially important for new developers: when they create a
new function, they don't necessarily know their way around the test
suite and documentation.

I propose writing TODO-comments in the code:

    # TODO test, document
    def myfun():
        pass

    # TODO document
    class MyClass:

        # TODO test, document
        def yourfun():
            pass

If you agree, I will write a small tool that extracts these todos:

    $ scripts/list_todos
    hkrc.py:
        myfun: test, document
        MyClass: test
        MyClass.yourfun: test, document

My proposal was this:

- everything else goes into the issue tracker (now todo file, later
  probably a heap)

How is your suggestion different from this? Do you suggest placing
notes about other minor missing things in the source code?

I'm not sure how I feel about that. My problem with that is the
following: in an issue tracking system, these issues can be discussed
easily and it is easy to follow how many issues we have, and to
categorize (tag) them. It is also harder to collect the todos from the
source code if we allow their broader usage. On the other hand, the
context of a note would be easier to grasp if it was placed in the
source code. I also realize that the issue tracking system is more
heavy-weigth than just writing a TODO-comment and developers don't
feel like writing an email about a minor thing they should just do.

We should think about pros and contras of the alternatives.
